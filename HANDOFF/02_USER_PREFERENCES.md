## ✅ 사용자가 좋아하는 톤·방식

- **정직** — 결과가 안 좋으면 안 좋다고 솔직히. 부풀리거나 우회하지 말 것.
- **풀어 설명** — 청중이 멀티모달을 모른다 가정. 용어·맥락 다 풀어서.
- **수치 + 평가 조건 명시** — "F1 0.97" 같은 수치만 쓰면 "어떤 시점에서 잰 건데?" 라고 물어봄. 항상 *"두 모달리티가 함께 입력되는 시점 2,161개 기준 F1 0.97"* 처럼 조건 명시.
- **결정해서 진행** — 사용자가 "둘 다 OK"라 하면 묻지 말고 합리적으로 결정.
- **직접 파일 확인** — 모르면 사용자한테 묻기 전에 파일을 먼저 뒤져보기. (특히 이전 대화나 결과)
- **간결한 보고** — 결과 요약은 표·짧은 문장. 장황한 설명 X.

## ❌ 사용자가 명시적으로 화낸 표현·행동

### 표현
1. **"사과 대 사과 비교"** — "갑자기 사과는 왜 나옴? 비유 빼고 정확히 뭐 대 뭐인지 명시"
2. **"불공정/공정 비교"** — 내부 디버깅 용어. 청중은 이게 뭘 의미하는지 모름. PPT엔 결과·결론만.
3. **"숨기는 게 아니라 솔직하게"** — "갑자기 뭘 숨기는데? 우리끼리 대화 톤이 그대로 PPT에 들어옴"
4. **빙빙 돌리며 같은 답 반복** — "내가 하나도 이해가 안 간다고 얘기하는데 똑같은 얘기만 하면 내가 이해를 하겠냐고"

### 행동
5. **PPT에 한 줄 스크립트 + 10분 분량이라 주장** — "이게 10분 분량이라고? 설명이 너무 부실"
6. **결과 수치 평가 조건 안 적음** — "0.9였다가 0.4면 왜 차이나는지 설명도 없고"
7. **백그라운드 작업 안전장치 호언했는데 작동 안 함** — Kaggle API 인증 거부로 폴링이 처음부터 작동 불가였는데 12시간 후에야 알아챔. 정직하게 사과한 뒤 정정.

## 🧠 사용자의 이해 수준 (대답할 때 참고)

- 딥러닝 개념은 알지만 멀티모달은 처음
- "타임스텝 vs 사이클 vs CycleState" 같은 데이터 구조를 여러 번 헷갈려함 → 항상 명확히 구분해서 말할 것
- 평가셋이 모델마다 다르다는 걸 이해하기 어려워했음 → 결과 보여줄 때 반드시 *"어느 시점 집합에서 측정한 거"* 명시

## 📐 작업 처리 방식

- **새 자료/도구 도입 시**: 사용자에게 사전 통보 + 짧게 이유 설명 ("이거 이걸로 갈게요, 왜냐면...")
- **실험 결과 분석 시**: 표로 비교 → 핵심 한 문장 → 시각자료
- **에러/한계 만났을 때**: 정직히 ("이건 못 함, 대신 이걸 시도하자")
- **사용자가 잠시 자리 비울 때**: 백그라운드 작업이 정말 작동하는지 미리 검증
- **PPT 만들 때**: 시각 QA(렌더링) 필수. 텍스트 잘림·겹침 체크

## 🎯 마감 관련

- 발표 임박 (정확한 날짜는 사용자에게 확인)
- 추가 학습은 안 함 (GPU 쿼터 거의 다 씀, 새 계정으로 한 번 더 돌렸지만 결과는 이미 나옴)
